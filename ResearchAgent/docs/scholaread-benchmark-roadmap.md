## 我们从 ScholaRead 借鉴什么

1. 低摩擦阅读入口：用户上来就能开始读，而不是先理解复杂配置。
2. 围绕论文理解的功能主轴：翻译、摘要、引用、问答都围绕“更快读懂”组织。
3. 明确的数据归档心智：用户知道自己的论文、标注、知识资产会沉淀下来。
4. 对学术工作流友好：引用、术语、上下文跳转这类动作都直接服务科研与写作。

## 我们应该超越什么

1. 不只做“辅助阅读器”，要做“可追问、可沉淀、可复用”的本地学术知识库。
2. 不只做单轮摘要，而是做基于原文上下文缓存的多轮深读与长期会话记忆。
3. 不只停留在 PDF 阅读，而是把论文、技术博客、网页报告统一纳入知识资产。
4. 不只帮用户看完一篇，而是帮用户形成主题、标签、追问、对比和复盘闭环。

## 分阶段路线

### Phase 1: 稳定的深读问答层

目标：让用户“读一篇论文时能持续追问，且不浪费上下文”。

1. 已完成：基于论文 / 解析内容的多轮聊天、Gemini cache 优先、会话持久化。
2. 下一步：把聊天回复里的页码引用和正文一样做成可点击跳转。
3. 下一步：增加“追问模板”，例如“讲算法增量”“讲实验可信度”“讲工程瓶颈”。
4. 下一步：支持“针对某个段落 / 图表发问”，减少用户手动复制上下文。

### Phase 2: 论文知识拆解层

目标：让用户更快抓住关键技术知识，而不是只看到一篇长文总结。

1. 自动生成“数据 / 算法 / 工程 / 风险”四栏知识卡。
2. 自动抽取论文中的关键实体：模型名、方法名、数据集、指标、系统组件。
3. 自动生成“本文相比基线的增量点”与“值得质疑的假设”。
4. 对 Infra 论文补充“实现链路图”“吞吐 / 成本 / 稳定性”三维解读。

### Phase 3: 长期知识库与用户粘性

目标：把一次阅读变成长期可检索、可复用的知识资产。

1. 建立主题图谱：按模型、方法、组织、任务、基础设施组件做交叉索引。
2. 建立个人标签和收藏体系：支持用户自己维护长期研究方向。
3. 对同主题论文自动聚类，给出“相关工作链”和“推荐下一篇”。
4. 增加“我的追问记录”和“我的观点沉淀”，形成个人研究笔记。

### Phase 4: 超越 ScholaRead 的智能交互层

目标：从“读论文工具”升级成“研究助理”。

1. 支持跨论文对比问答：例如比较两篇 RL / Agent 论文的算法差异与工程代价。
2. 支持生成组会摘要、技术分享提纲、复现实验 checklist。
3. 支持围绕某一主题自动生成“阅读路径”，把论文串成知识地图。
4. 支持把网页技术博客、开源发布说明、论文统一纳入同一问答上下文。

## 产品设计原则

1. 首页永远以“继续阅读 / 继续追问”为核心，不让用户先处理配置。
2. 任何复杂能力都优先嵌入阅读流和聊天流，而不是堆成独立功能页。
3. 核心 KPI 不是“解析篇数”，而是“用户更快理解关键技术点的速度”。
4. 数据沉淀优先于一次性炫技：所有摘要、标签、追问、观点都要可复用。
